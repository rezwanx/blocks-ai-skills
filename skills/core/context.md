# Project Context Skill

Defines how to create, read, and maintain `PROJECT.md` — the persistent project memory file that lives in the developer's project root.

`PROJECT.md` is committed to the project repo. Any Claude session in that project reads it at startup and skips pre-flight questions that are already answered.

---

## When to Read

**At every session start** — read `PROJECT.md` before asking any pre-flight questions.

If the file exists:
- Skip any pre-flight question whose answer is already recorded
- Use stored values (roles, schema names, agent IDs, etc.) directly in flows without re-asking
- State what context was loaded: _"I can see this project uses email/password login with email OTP MFA and has 3 schemas: Products, Orders, Users."_

If the file does not exist:
- Proceed with normal pre-flight questions
- Create it after the first feature is built (see When to Create below)

---

## When to Create

Create `PROJECT.md` after the **first feature** is completed, using answers gathered during that session's pre-flight questions.

Use this exact template:

```markdown
# PROJECT.md — [Project Name]

> Auto-generated by Claude Code. Update this file as the project grows.
> Claude reads this at session start to skip repeated pre-flight questions.
```

## When to Update

Update `PROJECT.md` immediately after any of the following:

| Event | What to update |
|-------|---------------|
| New schema created | Add row to Data Schemas table |
| New role created | Add row to Roles table |
| New agent created | Add row to AI Agents table with its ID |
| New KB folder created | Add row to KB Folders table with its item_id |
| Feature completed | Add row to Features Built table |
| Auth setting confirmed | Update Auth Configuration table |
| New project info known | Update Project Info table |

**Always update before ending a session** — do not leave newly created resources unrecorded.

---

## How It Reduces Pre-flight Questions

Before asking any pre-flight question from `core/decision.md` or a flow file, check `PROJECT.md` first:

| Pre-flight question | Check in PROJECT.md |
|--------------------|---------------------|
| Which login methods are enabled? | Auth Configuration → Login methods |
| Is MFA required? Which type? | Auth Configuration → MFA |
| Is CAPTCHA enabled? Which provider? | Auth Configuration → CAPTCHA |
| Which roles exist? | Roles table |
| Is this self-registration or admin-created? | Auth Configuration → Registration type |
| Which schemas exist? | Data Schemas table |
| Is there an existing agent? | AI Agents table |
| Which KB folder to use? | KB Folders table |
| Which embedding model was used? | KB Folders → Embedding Model |

If the answer is in `PROJECT.md`, **do not ask** — use it directly and confirm in your response:

```
// ✅ correct — use stored context, confirm it
Using the email/password login with email OTP MFA as configured in PROJECT.md.
If this has changed, let me know before I proceed.
```

---

## Schema IDs and Agent IDs

`PROJECT.md` is the only persistent place to store IDs returned by the API that are needed in future sessions. Always record:

- `agent_id` from `create-agent` response
- `item_id` from `create-kb-folder` response (this is the `kb_folder_id`)
- Schema names exactly as defined (needed for GraphQL operation naming)

Without these, future sessions must re-fetch via list APIs — avoidable overhead.

---

## File Location

`PROJECT.md` lives in the **developer's project root**, not in the skills repo.

```
my-project/              ← developer's app repo
├── PROJECT.md           ← project context (committed)
├── .env                 ← credentials (gitignored)
├── src/
└── ...
```

It is committed to version control. All team members and future Claude sessions benefit from it.

---

## What Not to Store in PROJECT.md

| Do not store | Reason |
|-------------|--------|
| `ACCESS_TOKEN` / `REFRESH_TOKEN` | Runtime secrets — `.env` only |
| `PASSWORD` | Credential — `.env` only |
| Full API responses | Too verbose — record only IDs and key values |
| In-progress work | Use task lists during the session, update PROJECT.md on completion |
